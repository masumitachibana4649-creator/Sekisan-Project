## 概要

このチャットでは、壁紙積算システムのPDF AI読取精度向上について、検討、実装、検証、Render上の実行エラー調査、処理時間計測、表ページ指定、解析状態表示、ドキュメント更新を行った。

主なテーマは以下。

- 部屋抽出の母集団を平面図中心から表ページ中心へ寄せる

- 表ページ候補、AI抽出、画面表示、CSV件数の差異確認

- バックグラウンド処理化に向けた土台実装
- `DOC/壁紙積算精度向上.md` の更新

### 1. 部屋抽出で使用している図面の確認

ユーザーから、現状では図面に描かれている部屋が全部抽出できていないため、以下の確認依頼があった。

### 2. 第1ステップの実装方針

ユーザーから以下方針で修正依頼があった。

### 13. ローカル・Renderでの動作確認と環境設定

ユーザーから、`.env` を修正したのでローカルで再確認してほしいとの依頼があった。

その過程で以下の問題が確認された。

- `http://127.0.0.1:8000/` が接続拒否
- Djangoで `DisallowedHost` が発生
- `127.0.0.1` を `ALLOWED_HOSTS` に追加する必要があった

その後、Render上では `サンプル図面５☆.pdf` は成功するが、`サンプル図面３☆.pdf` は失敗するという状況も確認された。

### 14. RenderのWorker Timeout

Renderで `サンプル図面５☆.pdf` の積算や再計算を行うと、以下のログが出た。

```text
[CRITICAL] WORKER TIMEOUT
[ERROR] Error handling request /projects/new/
[ERROR] Worker was sent SIGKILL! Perhaps out of memory?
```

原因として、OpenAI API呼び出しやPDF処理がGunicornのtimeoutを超えている可能性が高いと整理した。

対策として、RenderのStart CommandでGunicorn timeoutを一時的に延長する案を確認した。

例:

```text
--timeout 300
```

また、`OPENAI_VISUAL_TABLE_PAGE_DETECTION` が `.env` にない場合はデフォルトでfalse扱いになることも確認した。

## Render関連メモ

### 発生した主なエラー

```text
Internal Server Error
```

```text
[CRITICAL] WORKER TIMEOUT
[ERROR] Worker was sent SIGKILL! Perhaps out of memory?
```

### 一時対策

Gunicorn timeoutを延長。

例:

```text
--timeout 300
```

これにより、一部ケースではInternal Server Errorを回避できた。

### 注意点

Gunicorn timeoutやSIGKILLでプロセスが強制終了した場合、Djangoの例外処理まで到達できないため、失敗状態や計算時間が保存されない可能性がある。
