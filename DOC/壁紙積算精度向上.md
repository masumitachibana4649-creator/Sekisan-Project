# 壁紙積算精度向上まとめ

## 1. 現在の壁紙積算のまとめ

### 1.1 Input情報（PDF、対象図面のページ）

壁紙積算の主な入力は、ユーザーがアップロードする建築図面PDFと、画面で指定する対象ページである。

画面で指定するページ情報は以下。

| 入力項目 | 用途 |
| --- | --- |
| 1F平面図 | 部屋抽出、階判定、寸法・開口推定の根拠 |
| 2F平面図 | 部屋抽出、階判定、寸法・開口推定の根拠 |
| 3F平面図 | 3階がある場合の部屋抽出、階判定 |
| 展開図開始頁 | 壁面、開口部、面別情報の根拠 |
| 展開図終了頁 | 壁面、開口部、面別情報の根拠 |
| 1F天井伏図 | 平面図の室名候補補完、天井面積・天井高補助 |
| 2F天井伏図 | 平面図の室名候補補完、天井面積・天井高補助 |
| 3F天井伏図 | 3階がある場合の室名候補補完 |

現在の設計では、平面図・天井伏図・展開図に加えて、PDF内から自動検出した表ページもAI読取対象に追加する。

表ページとして検出する対象は以下。

| 図面名 | 主な用途 |
| --- | --- |
| 居室区画面積表 | 部屋候補、階、面積の主根拠 |
| 床面積表 | 部屋候補、階、面積の主根拠 |
| 室内仕上表 | 仕上、壁・天井情報、部屋候補補助 |
| 内部仕上表 | 仕上、壁・天井情報、部屋候補補助 |
| 建具表 | 開口部、建具寸法の補助根拠 |

サンプル図面で確認した代表的なページ指定は以下。

| PDF | ページ指定 |
| --- | --- |
| サンプル図面3 | 1F平面図=9P、2F平面図=10P、展開図=11-12P、1F天井伏図=6P、2F天井伏図=6P、床面積表=14P、居室区画面積表=22P |
| サンプル図面5 | 1F平面図=5P、2F平面図=5P、展開図=16-22P、1F天井伏図=8P、2F天井伏図=8P、室内仕上表=4P、建具表=10-13P |

### 1.2 計算処理の流れ

現在の処理は、大きく以下の流れで実行される。

1. PDFアップロード
   - 画面からPDFをアップロードする。
   - ローカルでは `media/drawings/`、RenderではSupabase Storageに保存する。
   - PDFヘッダ、MIMEタイプ、ファイルサイズを検証する。

2. ページ指定の検証
   - `1F平面図` などのページ番号を検証する。
   - `ー`、空欄、`0` は未指定として扱う。
   - 展開図は開始頁・終了頁の整合性を確認する。

3. 表ページ検出
   - `pypdf` でPDF全文テキストを抽出し、表ページ名を検索する。
   - 文字抽出が崩れるPDFでは、文字化けパターンから軽量に表ページ候補を補完する。
   - 画像/OCR寄りのAI表ページ判定も実装済みだが、Renderの同期処理ではtimeoutリスクがあるためデフォルトOFF。
   - 必要な場合のみ `OPENAI_VISUAL_TABLE_PAGE_DETECTION=true` でONにする。

4. 表ページから部屋候補抽出
   - `居室区画面積表` と `床面積表` を優先して、階・部屋名・面積を抽出する。
   - `室内仕上表` / `内部仕上表` からも候補抽出を試みる。
   - `収納` は同一階で `収納 一式` に集約する。
   - 同一階同名の `CL` などは `CL 1`、`CL 2` のように個別化する。
   - `UB`、浴室、バルコニー、ポーチ、屋外は壁紙対象外候補として除外する。

5. AI読取対象PDFの作成
   - 指定された平面図、天井伏図、展開図範囲を抽出する。
   - 自動検出した表ページも追加する。
   - 重複ページは除外する。
   - 抽出したページだけを一時PDFにしてOpenAIへ渡す。

6. OpenAIによる部屋情報抽出
   - AIへ以下を抽出させる。
     - 部屋名
     - 周長
     - 天井高
     - 開口部面積
     - 天井面積
     - 1面から4面の壁面積、開口部面積
   - ロール本数、ロス率込み面積、金額はAIに計算させず、アプリ側で計算する。

7. AI応答の検証と補完
   - AI抽出部屋と表ページ候補を照合する。
   - 未表示候補は `missing_rooms` として抽出失敗追加部屋を作成する。
   - 表ページ候補に面積がある場合、抽出失敗追加部屋にも天井面積として反映する。

8. DB保存前の安全化
   - 部屋名、備考をDBの最大文字数以内に丸める。
   - 面積、周長、開口部などの数値をDB桁数内に丸める。
   - 負数、不正値、無限大は0に補正する。

9. 壁紙量・ロール本数・金額計算
   - 壁面積、天井面積、開口部控除、ロス率、ロール面積からアプリ側で算出する。
   - 壁紙別合算方式と部屋別積上方式の両方を計算する。

10. 画面表示・CSV出力
    - 部屋別明細、壁紙別集計、合計面積、ロール本数、金額を表示する。
    - CSV出力では部屋別明細と壁紙別集計を出力する。

### 1.4 使用しているAIの情報

AI読取にはOpenAI APIを使用している。

| 項目 | 内容 |
| --- | --- |
| API | OpenAI Responses API |
| クライアント | `openai.OpenAI` |
| 使用モデル | 環境変数 `OPENAI_PDF_ANALYSIS_MODEL`。未指定時は `gpt-4o` |
| 入力 | AI読取対象ページだけを抽出したPDF、解析プロンプト |
| 出力形式 | JSON Schemaによる構造化出力 |
| 主な出力 | `rooms`、`missing_rooms`、`warnings` |
| AIに計算させないもの | ロール本数、ロス率込み面積、金額 |

AIに要求している `rooms` の主な項目は以下。

| 項目 | 内容 |
| --- | --- |
| `name` | 部屋名 |
| `perimeter_m` | 周長 |
| `height_m` | 天井高 |
| `opening_area_m2` | 開口部面積 |
| `ceiling_area_m2` | 天井面積 |
| `wall_surfaces.face_1` - `face_4` | 面別の壁幅、壁面積、開口部面積 |
| `confidence` | AI信頼度 |
| `evidence` | 根拠説明 |

表ページ画像/OCR判定については、同じOpenAI APIを利用する補助機能として実装されている。ただしRenderの同期処理ではタイムアウトリスクがあるため、現在はデフォルトOFFである。

## 2. 現在の問題点

### 2.1 PDF文字抽出が崩れる図面への弱さ

PDFによっては、`pypdf` の `extract_text()` で日本語が文字化けする。

その結果、以下が起きる。

- 図面名検索で表ページを検出できない。
- 表ページから部屋候補を作れない。
- 平面図・天井伏図上の室名候補件数が実際より少なくなる。
- AIへ渡す候補リストが不足し、部屋数が減る。

PDF5では、通常の文字検索では表ページが検出できなかったため、文字化けパターンによる軽量検出を追加した。

### 2.2 画像/OCR表ページ判定は精度に有効だが同期処理では重い

OpenAIにPDF全体を渡して表ページを視覚判定する機能は、文字化けPDFに有効である。

一方で、Renderの通常HTTPリクエスト内で実行すると以下の問題がある。

- OpenAI呼び出しが2回になる。
- PDF全体の判定と積算本体の両方で時間がかかる。
- Gunicorn worker timeoutが発生する。
- Renderログでは `WORKER TIMEOUT`、`SIGKILL` が発生する。

そのため、現在は画像/OCR表ページ判定をデフォルトOFFにしている。

### 2.3 初回計算と再計算で最適な処理が異なる

初回計算では表ページ情報がまだメモに存在しない。

再計算では、初回成功時のメモに以下のような情報が残る。

```text
室内仕上表=4P、建具表=10P、建具表=11P
```

現在は再計算時にこの情報を再利用し、重い表ページ再検出を避けている。

ただし、積算失敗後の再計算ではメモに表ページ情報がない場合があり、その場合は初回計算相当の処理になる。

### 2.4 表ページ候補が作れないPDFでは母集団が不足する

表ページ自体を検出できても、表の中身の文字抽出が崩れている場合、`階・部屋名・面積` の候補リストを作れない。

この場合、AIは表ページ画像を見て判断するが、アプリ側で候補全件を強制的に渡すことはできない。

結果として、以下が起きる。

- AI抽出部屋数が少ない。
- `missing_rooms` に依存する。
- 表ページ上の23件などの母集団へ寄せきれない。
- 説明文の件数と画面表示件数に差が出やすい。

### 2.5 AI応答は毎回揺らぐ

OpenAIのPDF読取結果は同じPDFでも完全には固定されない。

揺らぎやすい項目は以下。

- 抽出される部屋数
- `missing_rooms` の内容
- 部屋名の表記
- 開口部面積
- 面別壁面積
- 備考・根拠文の長さ

Render/PostgreSQLでは文字数や数値桁数の制約がSQLiteより厳しいため、AI応答をDB保存前に丸める処理を追加済みである。

### 2.6 件数内訳と実際の作成結果が一致しない場合がある

説明文には以下の件数が表示される。

- AI抽出件数
- 抽出失敗追加件数
- 表示合計
- 表ページ候補件数
- 未表示候補

ただし、現状では以下の理由により実際の画面行数とずれる場合がある。

- AIの `missing_rooms` と表候補由来の追加部屋の扱いが分かれる。
- 同名部屋の集約・個別化ルールが完全には統一されていない。
- 収納一式、CL、PS、吹抜などの扱いが図面ごとに変わる。
- AI抽出後にDB保存前補正や抽出失敗追加が入る。

この件は、現時点では「置いておく」扱いになっている。

### 2.7 同期HTTP処理の限界

現在はPDFアップロードまたは再計算ボタン押下のHTTPリクエスト内で、PDF解析とOpenAI呼び出しを完了させている。

この方式には以下の制限がある。

- Render/Gunicornのtimeoutに影響される。
- PDFが重いと画面応答が遅い。
- OpenAI応答が遅いと500やworker timeoutになりやすい。
- 長期的にはバックグラウンドジョブ化が望ましい。

### 2.8 精度向上と速度のトレードオフ

精度を上げるには以下が有効である。

- 表ページ画像/OCR判定を使う。
- 表ページ全体をAI読取対象に含める。
- 候補リスト全件をAIに渡す。
- 建具表も開口部推定に使う。

一方で、これらは処理時間とOpenAI呼び出し量を増やす。

現在はRenderのtimeout対策として、以下の方針にしている。

- 通常の同期計算では軽量検出を優先。
- 再計算では既存メモの表ページ情報を再利用。
- 画像/OCR表ページ判定は明示ONの時だけ利用。

## 3. 今後の改善候補

1. PDF解析をバックグラウンドジョブ化する。
   - HTTP request timeoutを避ける。
   - 重い画像/OCR判定を安全に使える。

2. 表ページOCR結果をDBまたはファイルにキャッシュする。
   - 初回検出結果を再計算で安定利用できる。
   - 再計算時のOpenAI呼び出しを削減できる。

3. 表ページの画像OCRから候補リストを構造化する。
   - `階・部屋名・面積` をAIまたはOCRで表形式抽出する。
   - pypdf文字化け時でも候補母集団を確保できる。

4. 件数内訳の算出元をDB作成後の実データへ寄せる。
   - 説明文、画面、CSVの件数差異を減らす。

5. 図面タイプ別の処理モードを分ける。
   - 通常PDF
   - 文字化けPDF
   - 表ページ重視PDF
   - 展開図重視PDF

6. AI結果の監査ログを保存する。
   - 入力ページ
   - 検出表ページ
   - AI応答JSON
   - DB作成結果
   - 件数差異

7. 表示上の「推定値」「抽出失敗」「表ページ反映」の区分をより明確にする。
