# PDF解析プロンプト整理 チャット記録

## 概要

2026-06-14に、壁紙積算のPDF解析AIへ渡すプロンプト整理と、PDF解析のバックグラウンドジョブ化について検討した。

最終結論は以下。

- PDF解析プロンプトは、挙動を大きく変えずにブロック化した。
- 小さな最終チェックブロックを追加した。
- 階段条件、重複統合、統合禁止ルールの追加は、サンプル図面5で出力が不安定化したため採用しなかった。
- 一度DBキュー方式のバックグラウンドジョブ化を実装したが、RenderのBackground Workerが無料で使えない構成だったため破棄した。
- GitHubの `main` は `ff96241 Restore flat analysis prompt rules` に戻した。

## 1. プロンプト整理の依頼

最初の目的は、PDF解析をバックグラウンドジョブ化する前に、複雑になった `estimator/pdf_analysis.py` の `_analysis_prompt()` を整理することだった。

対象ファイル:

- `DOC/壁紙積算精度向上.md`
- `DOC/チャット記録.md`
- `estimator/pdf_analysis.py`

ユーザーからの依頼:

> まず 挙動を変えずにプロンプトをブロック化してください

対応:

- `_analysis_prompt()` の長い文字列をブロック定数へ分割した。
- 既存ルールの内容はなるべく変更しない方針にした。

追加した主なブロック:

- `ANALYSIS_PROMPT_INTRO`
- `ANALYSIS_PROMPT_TARGETS`
- `ANALYSIS_PROMPT_RULES`

## 2. ブロック化後の確認

サンプル図面5で確認した結果、プロンプトブロック化後も部屋抽出件数は大きく崩れなかった。

確認時の傾向:

- 修正前: 16件前後
- ブロック化後: 16件前後
- 処理時間はやや短くなったが、AI応答の揺らぎの範囲と判断した。

## 3. 最終チェックブロック追加

ユーザーからの依頼:

> 最終チェックブロック追加 を小さく入れて

対応:

- `ANALYSIS_PROMPT_FINAL_CHECK` を追加した。
- `_analysis_prompt()` の最後に追加した。
- 役割は、出力直前に以下を再確認させること。

最終チェック内容:

- 平面図・天井伏図・展開図・表ページの主要部屋を漏らしていないか。
- 浴室、UB、バルコニー、ポーチ、屋外などを対象外にしているか。
- ロール本数、金額、ロス率込み面積をAI側で計算していないか。
- 推定値には根拠を記載しているか。

検証結果:

- サンプル図面5で17件前後となり、部屋漏れ確認の効果が見られた。
- この変更は採用した。

## 4. 階段条件の追加と破棄

一時的に以下の条件を追加した。

> 階段の壁高さは、展開図や断面図で階高が読める場合は標準天井高ではなく階高・断面寸法を優先してください。

その後、以下のように修正した。

> 階段の壁高さ補正は階段室の height_m だけに適用し、他の部屋候補の統合・削除理由にしないでください。

しかしサンプル図面5の出力で、部屋数や部屋候補の扱いが不安定になった。

最終判断:

- 階段条件は削除した。
- 83相当の状態へ戻すことにした。

## 5. 重複統合と統合禁止ルールの検討

プロンプト内の重複統合を試みたが、出力が不安定化した。

追加検討した統合禁止ルール:

> 平面図または展開図で別室名・別区画として確認できる居間、台所・食堂、LDK、寝室、洋室、和室、畳室は、近接していても統合せず別部屋として扱ってください。

検証結果:

- 抽出件数が増減した。
- 建具表の読取ページ範囲が揺らいだ。
- 既存の83相当より安定しなかった。

最終判断:

- 重複統合整理は採用しない。
- 統合禁止ルールも採用しない。
- 最終状態は「プロンプトブロック化 + 最終チェックブロック追加」とする。

## 6. 83相当への復帰

ユーザーからの依頼:

> 83に戻してください
> ・プロンプトブロック化
> ・最終チェックブロック追加
> ・重複統合なし
> ・階段条件なし

対応:

- 階段条件を削除した。
- 重複統合や統合禁止の追加ルールを削除した。
- ブロック化と最終チェックブロックは維持した。

この状態で、サンプル図面5は従来に近い挙動へ戻った。

## 7. DBキュー方式のバックグラウンドジョブ化

PDF解析が長時間かかるため、DBキュー方式でバックグラウンドジョブ化を検討した。

一時的に実装した内容:

- `PdfAnalysisJob` モデル追加
- `PdfAnalysisJob` 管理画面追加
- `0016_pdfanalysisjob.py` マイグレーション追加
- `project_create` と `project_recalculate` を即時解析からジョブ登録へ変更
- `_run_pdf_analysis_job()` を追加
- `run_pdf_analysis_jobs` management command追加
- `project_detail.html` に「PDF解析中です。」表示と自動更新を追加
- `render.yaml` にBackground Workerを追加

想定運用:

```bash
python manage.py run_pdf_analysis_jobs
```

## 8. Render無料運用との不一致

DBキュー方式にした後、ユーザーがRender画面を確認した。

現在のRender構成:

- `Sekisan-Project`: Web Service
- `Sekisandb`: PostgreSQL
- Background Workerは存在しない

そのため、画面は「PDF解析中です。」のまま30分以上完了しなかった。

原因:

- Web ServiceはジョブをDBへ登録するだけ。
- Background Workerがないため、pendingジョブを処理するプロセスが存在しない。

Background Worker作成画面では、Freeが表示されず、Starter `$7/month` からだった。

結論:

- Render無料運用では、Background Worker方式を採用しない。
- DBキュー方式の実装は破棄する。

## 9. コミットの巻き戻し

ユーザーからの依頼:

> 無料でしたいのでPDF分析をバックグラウンドジョブとして実行するのを破棄して
> 1つ前に戻したいです。
>
> GitHUb Commit dbf8bee を破棄して、Commit ff96241に戻すことはできますか？

確認結果:

```text
dbf8bee Run PDF analysis as background jobs
ff96241 Restore flat analysis prompt rules
```

対応:

```powershell
git reset --hard ff96241
git push --force-with-lease origin main
```

結果:

```text
+ dbf8bee...ff96241 main -> main (forced update)
```

最終状態:

- ローカル `main` と `origin/main` は同期済み。
- HEADは `ff96241 Restore flat analysis prompt rules`。
- DBキュー方式のコミット `dbf8bee` は `main` から破棄済み。

## 10. 現在の採用状態

現在採用している内容:

- PDF解析は同期HTTP処理のまま。
- `_analysis_prompt()` はブロック化済み。
- 最終チェックブロックは追加済み。
- 階段条件はなし。
- 重複統合整理はなし。
- 統合禁止追加ルールはなし。
- DBキュー方式のバックグラウンドジョブ化はなし。

## 11. 今後の注意点

無料運用を続ける場合、PDF解析の長時間化対策は同期処理の範囲で行う必要がある。

候補:

1. 表ページ指定を正確に入力する。
2. AIへ渡すページ数を必要最小限にする。
3. 画像/OCR表ページ判定は通常OFFにする。
4. プロンプトは83相当の安定状態を基準にする。
5. 有料運用を許容できる段階で、Background WorkerまたはCron Jobを再検討する。
